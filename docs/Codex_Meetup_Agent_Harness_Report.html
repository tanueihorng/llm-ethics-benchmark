The short version, before the detail: most people use a coding agent. This project engineers the environment the agents run in — the rules, the memory, the guardrails, and the verification gate — so that two different agents from two different vendors can pick up the same research project, days apart, and not break it. Everything below is grounded in real files in the repository; nothing is aspirational.

A primer on agent & harness engineering

A shared vocabulary first, so the rest of the report lands. If you already live in this world, skim — the one idea that matters is the ladder from prompt → context → harness.

The three eras of getting useful work out of a model

The industry has moved through three phases of practice. Each one didn't replace the last — it wrapped it.

What a "harness" actually is

A production-grade agent harness is usually described as five layers. Keep these in mind — Section 4 shows that this repository implements all five, by hand, in plain files:

🛠️
Tool orchestration

What the agent can do, and how calls are routed and sequenced.

🔁
Verification loops

Plan → execute → verify. Deterministic checks and review gates before a change is trusted.

🧠
Context & memory

What persists between turns and sessions, and where the source of truth lives.

🛡️
Guardrails

What the agent must never do — privacy, safety, integrity boundaries.

📈
Observability

A trail of what happened, so a human (or the next agent) can audit it.

🎯
The point

Reliability comes from the harness, not from a smarter prompt. The model is a component; the harness is the system.

The primitives every modern coding agent now shares

Codex, Claude Code, Cursor, Gemini CLI and ~30 others have converged on the same handful of building blocks. This convergence is why a dual-agent setup is even possible — the two agents speak the same dialect.

PrimitiveWhat it isWhere it lives
Instruction fileA "README for agents" — house rules, conventions, where to look.AGENTS.md / CLAUDE.md
SkillsOn-demand procedures the agent loads only when a task matches — keeps the base prompt small.SKILL.md files
SubagentsSpecialised, context-isolated helpers spawned for a sub-task (explore, audit, review).TOML / config files
HooksCode that fires on lifecycle events (before a tool runs, before compaction, on stop) for policy & logging.hooks.json / config.toml
MCPModel Context Protocol — a standard socket for plugging external tools/data into any agent.server configs
The thesis of this report This project doesn't just use these primitives — it assembles them into a coherent harness whose job is research integrity: never mutate raw experimental data, never leak harmful prompt text, never let two agents drift out of sync, and never trust a claim that the repo can't verify.

The Codex landscape (2025 → 2026)

Since the meetup is hosted by the Codex team, here is the lay of the land — accurate as of mid-2026 — and exactly which Codex features this repo leans on.

Codex, the short history

Apr 2025Codex CLI launches — an agentic coding agent in the terminal: reads the repo, edits files, runs tests, commits.
May 2025Codex Cloud research preview, powered by codex-1 (an o3 variant tuned for software engineering).
Dec 2025Skills land — a SKILL.md file under ~/.agents/skills/, auto-loaded when a task matches.
Mar 20262M+ weekly active users; Codex Security ships; positioned as an enterprise agent platform.

Codex now runs across one continuous surface — the ChatGPT web app, the CLI, a desktop app (macOS/Windows), IDE integrations, and the cloud — with limits shared across all of them. That "same agent, many doorways" model matters for this project: work can start in the CLI on a laptop and continue in the cloud, against the same repo.

AGENTS.md — the open standard underneath it all

AGENTS.md was released by OpenAI in August 2025, then handed to the Linux Foundation's Agentic AI Foundation in late 2025. By May 2026 it was in 60,000+ repositories and read by 30+ agents (Codex, Claude Code, Copilot, Cursor, Gemini CLI, Aider, Zed, Windsurf, Devin…). It is deliberately boring: plain markdown, no required fields, no YAML front-matter. In a monorepo the nearest file to the edited file wins.

Why this matters for a dual-agent setup Because AGENTS.md is a shared standard, Codex and Claude Code can read the same house rules. This repo keeps a CLAUDE.md for Claude and an AGENTS.md for Codex that are byte-identical below the title line — and the harness has a check that fails the build if they ever drift apart (see §4). One source of rules, two readers.

Codex subagents & hooks — the exact mechanisms used here

Two Codex features deserve a close look because the harness is built directly on them:

Subagents  .codex/agents/*.toml

Each TOML file defines one custom agent with name, description, developer_instructions. Codex spawns them in parallel, isolates their context, and collects results into one answer. Global knobs: max_threads (default 6), max_depth (default 1 — no deep nesting). Subagents inherit the parent's sandbox. Trade-off: more tokens than a single agent, bought for cleaner separation.

Hooks  .codex/hooks.json

Lifecycle handlers configured per-repo. A hook declares a matcher and a command; it receives data on stdin and signals via exit code (0 allow, 1 block). Used for audit logging and policy enforcement — e.g. a PreToolUse hook that inspects a Bash command before it runs.

# .codex/hooks.json — the real shape of a Codex hook
{
  "PreToolUse": [{
    "matcher": "Bash|Edit|Write|apply_patch",
    "hooks": [{ "type": "command",
      "command": "python3 .codex/hooks/guard_artifact_paths.py" }]
  }],
  "PreCompact": [ /* refresh the handoff before context is compacted */ ],
  "Stop":      [ /* run a lightweight finish check when a turn ends */ ]
}

That is precisely the set of three hooks this project ships. They are reminders, not blockers — the authoritative gate is a separate command (§4). It's a deliberate design choice: hooks nudge, the gate decides.

The research project this harness serves

You can't appreciate the harness without knowing what it protects. The agents aren't writing a web app — they're running a controlled scientific experiment where the data must stay trustworthy.

When you shrink a language model to 4‑bit to make it cheap to run, does it become less safe — or just less capable? And can you even tell the two apart?

That is the research question of CCDS25-1136, a final-year project at NTU's College of Computing and Data Science. The study takes three small instruction-tuned models, makes a 4-bit (NF4) copy of each from identical weights, and measures both copies along three axes — then asks whether any safety change is real alignment movement or just collateral capability loss.

Qwen 4BQwen3-4B → NF4ASRrefusal rateaccuracy
Llama 3.2 3BLlama-3.2-3B-Instruct → NF4ASRrefusal rateaccuracy

The finding — and why it's the harness's best demo

The project's headline result is also the cleanest illustration of why the harness exists. The first-pass HarmBench scorer was a fast regex. Validating it against the official 13B HarmBench classifier (run as a judge model on the cluster) showed the regex over-counts attack success — agreement was poor on the Qwen pairs (Cohen's κ ≈ 0.19–0.37) and only moderate on Llama (κ ≈ 0.69–0.79).

Decision D16 · the moment the harness earned its keep Switching to the judge as the primary scorer moved the study's only statistically significant safety regression from one model to another (from Qwen 4B under the regex to Qwen 1.7B under the judge). Under the judge, 4-bit quantization never reduces harmful compliance in any pair. A scoring layer changed the conclusion — but because raw model outputs are immutable evidence, nothing had to be re-run. The correction was a new sidecar, not a mutation.

The label is never altered by significance; a second "evidence_status" layer carries the statistical confidence separately (Decision D19). That separation is itself a guardrail against over-claiming — a recurring theme.

4

The repo-native harness, layer by layer

This is the centrepiece. Everything an agent needs to work safely on this project is a file in the repo — versioned, diffable, reviewable. The harness turns the project's prose rules into machine-checkable contracts.

The repository at a glance

Before the control plane, here is the whole project laid out by role. Four zones: the agent operating system (the harness itself), the research core (the science), the inputs & compute, and the evidence & deliverables. The harness governs the left; experimental evidence flows one-way into the right and is never mutated.

Layer ① — One source of truth, enforced

The rule the whole project hangs on: docs/PROJECT_LOG.md is the only file that records "what's happening now." Every other doc is reference or archive. Every change to the repo appends one row to its changelog — When | Files | Change | Why | Report? | Who — newest first. The Who column is literally how you see the dual-agent split (Section 6). This is the harness's observability layer.

Layer ② — Bounded context instead of a giant prompt

Rather than one bloated instruction file, the harness keeps the base lean and pulls in detail on demand:

MechanismFilePurpose
Task packetdocs/agent_tasks/T21-…mdA small, self-contained brief for one task: objective, likely files, what's allowed, what's forbidden, done-criteria.
Skill.agents/skills/fyp-report-audit/SKILL.mdA reusable procedure loaded only when the work matches — e.g. auditing report claims, adding a second judge.
Startup packetfyp_cli.py agent-start --task … --agent …Assembles the live status + the chosen task packet + the chosen subagent profile into one focused briefing.

Four skills exist today — fyp-harness-maintenance, fyp-report-audit, fyp-second-judge, fyp-meetup-brief — each with a clear scope, start step, guardrails, and finish step.

Layer ② (cont.) — Five context-isolated subagents

The five .codex/agents/*.toml profiles are all read-only reviewers by design — they audit, they don't mutate, and the main agent runs the gate:

🔐 fyp-artifact-guardian

Evidence-chain auditor: checks raw-artifact immutability, the SHA-256 manifest, and redaction leaks.

⚖️ fyp-judge-reviewer

Reviews the HarmBench judge layer — confirms the official classifier stays primary, checks the agreement analysis.

📄 fyp-report-auditor

Reviews report prose for stale claims, judge-vs-regex mistakes, and missing verification.

🖥️ fyp-tc1-ops

Cluster-safety reviewer: confirms compute is routed via sbatch and offline env vars are preserved.

🎤 fyp-meetup-story

Turns repo evidence into a human-readable workflow narrative — the agent that helped scope this very report.

Why isolate them?

An auditor with a clean, narrow context gives a sharper second opinion than the same model carrying the whole working session in its head.

Layer ④ — The gate: make agent-check

This is the verification loop made concrete. Eight checks, each reading the same artifact_policy.yaml. Any fail stops the build. What it enforces, in plain terms:

CheckFails when…
doc-syncAGENTS.md and CLAUDE.md drift apart below the title line.
log-updatedThe repo has changes but PROJECT_LOG.md isn't among them — no silent edits.
immutable-artifactsA raw raw.jsonl/summary.json hash no longer matches the manifest (tolerates absent files in fresh clones, fails on mutation).
redaction-scanRaw prompt/response/behavior text leaks into any doc, hook, or agent profile.
stale-textOutdated claims survive (e.g. "results pending", an old scorer being called primary).
report-freshA report-worthy source changed but the report docx wasn't regenerated.
whitespacegit diff --check finds trailing-whitespace noise.
pytestAny test in the full suite fails.
The detail that sells it There's a test that the harness itself works (make harness-eval): it builds a tiny throwaway repo, deliberately breaks each rule — mutates a raw artifact, drifts the two instruction files, plants a redaction leak — and asserts the gate catches each one. The guardrails are themselves under test. That's harness engineering taking itself seriously.

Memory: state lives in the repo, not the chat

The single biggest idea to communicate at the meetup. Chat history is volatile and private to one agent. This project pushes durable state out of the conversation and into versioned files any agent can read.

How the auto-memory and the repo cooperate

The agent does keep a small private memory (user preferences, a project-state snapshot, file pointers). But it is deliberately framed as an index, not an authority: it carries warnings like "the source of truth is always the repo — don't trust memory numbers." If a memory note names a file or a result, the rule is to verify it against the repo before acting. The harness even scans for stale prose so that out-of-date claims can't quietly persist.

The one-liner for the meetup "My agents' memory isn't in the chat — it's in the repository. PROJECT_LOG.md is the brain, Git is the audit log, and a hashed manifest makes the raw data tamper-evident. A brand-new session — even on a different vendor's agent — reads four files and is fully oriented."

The dual-agent workflow (Codex & Claude)

Two agents, two vendors, one repo. They don't talk to each other directly — they collaborate through the repository, the same way two humans collaborate through a shared codebase and a changelog.

How the labour actually divided — from the changelog

This isn't a theory; it's visible in the Who column of the changelog. Across the recorded history, the split looked like:

Claude — the analyst & writer

Judge-validation analysis and the McNemar/CI evidence layer; the two-layer reporting scheme; report prose; drafting the supervisor email; refreshing handoffs; landing and verifying Codex's uncommitted work before committing it.

Codex — the harness builder & auditor

The entire on-demand agentic layer (skills, subagents, hooks); the architecture diagrams; converting in-place rescoring into immutable sidecars after a near-miss; pre-push hardening audits; recording cluster submissions and the QoS concurrency discovery.

A real handoff, captured in one changelog row On one occasion Claude committed Codex's uncommitted work for it — syncing AGENTS.md/CLAUDE.md, applying a report appendix edit, regenerating the docx — but only after verifying the diffs matched what Codex's own changelog row claimed. That is the whole model in miniature: one agent implements, the other reviews against the recorded intent and merges. Trust, but verify from the repo.

Why split by vendor at all?

Two honest reasons, both worth saying out loud at the meetup: (1) different agents have different strengths and quotas, and routing work to the better-suited one is just good economics; (2) a second agent is a second pair of eyes — a reviewer that didn't write the code catches what the author misses. The harness is what makes the split safe: because all state is in the repo and the gate is vendor-neutral, it genuinely doesn't matter which agent picks up the work next.

How to explain it to peers

Ready-to-use framing for the room. Lead with the idea, back it with one concrete file, and let people pull on the thread.

The 30-second version

Say this "I run two coding agents — Codex and Claude Code — on the same research repo. Instead of trusting either one's memory, I made the repository the memory: one log file is the source of truth, the raw experimental data is hashed and immutable, and there's a single make agent-check gate that fails if the two agents drift, if data gets mutated, or if a harmful prompt leaks into a doc. It's harness engineering applied to research integrity — built entirely out of Codex's own primitives: AGENTS.md, skills, subagents, and hooks."

Five talking points, each with a concrete hook

1 · Memory lives in the repo

"A new session reads four files and is oriented." → open PROJECT_LOG.md and HANDOFF.md.

2 · Rules are executable, not just prose

"Prose rules rot. I compiled mine into checks." → run make agent-check; show artifact_policy.yaml.

3 · Two vendors, one standard

"AGENTS.md ≡ CLAUDE.md, enforced by a drift check." → show the doc-sync check failing on a deliberate edit.

4 · Evidence is immutable

"I changed the conclusion twice and never touched the data." → show the sidecar files + the SHA manifest.

5 · The guardrails are tested

"My harness has a test suite that proves it catches bad states." → run make harness-eval.

The hook for Q&A

"Hooks remind; the gate decides. I never let a hook silently block — a human (or the gate) stays in the loop." Invites the inevitable autonomy debate.

A 3-minute live demo, if you get a laptop

  1. Open a fresh terminal: python fyp_cli.py agent-status — show how the repo introduces itself.
  2. Open docs/PROJECT_LOG.md §4 changelog — point at the Who column alternating between Claude and Codex.
  3. Make a deliberate bad edit (drift CLAUDE.md, or add "RESULTS PENDING" to a doc) → run make agent-check → watch the relevant check go red.
  4. Show .codex/hooks.json and one subagent TOML — "these are stock Codex features; I just composed them."
  5. Close with Fig 9's story: the judge correction that moved the headline result without re-running a single experiment.

Anticipated questions — and honest answers

They'll askAnswer
"Isn't this over-engineering for a student project?"The over-engineering is the project's risk control: it's scientific data. The harness is small (a policy file, a check script, a few generated docs) and it's already caught a real near-miss (in-place rescoring) and a scorer error that changed the headline finding.
"Why two agents instead of one good one?"Different strengths and quotas, plus a genuine second reviewer. The harness makes the choice of agent irrelevant to correctness — that's the whole point.
"Do the hooks enforce anything?"No — hooks are reminders by design. The authoritative gate is make agent-check, run explicitly. Keeping enforcement in one visible place is deliberate.
"How is this different from a good README + CI?"It largely is that — which is the point. The novel part is treating agents as first-class repo citizens: bounded task packets, context-isolated reviewer subagents, an immutable-evidence contract, and a vendor-neutral instruction standard, all checked together.
