Orientation

01The project in five minutes

Question. When a compact language model (1.7–7.2 B parameters) is quantized to 4-bit so it fits on consumer hardware, do the safety changes people report reflect a genuine alignment shift — or are they artefacts of capability loss, invalid scoring, or truncated generation?

Design. Matched pairs: the baseline and the quantized member of each pair load from the identical weights, with bitsandbytes quantization applied on the fly at load time. Nothing else differs — same tokenizer, same chat template, same prompts, same seed, same decoding. So any measured delta is attributable to quantization alone.

What was measured. Harmful compliance (HarmBench, ASR), over-refusal on benign prompts (XSTest), and general capability (MMLU subset + ARC-Challenge), for five pairs across four families, at three precisions (fp16, INT8, NF4), at two generation budgets (an initial 128-token run, then the full re-run at HarmBench's own standardized 512-token budget, which is the primary study).

The headline, at the 512-token primary budget

  • No pair shows a significant increase in harmful compliance under direct requests. Not one of the 20 primary contrasts on the ASR axis survives the Benjamini–Hochberg multiplicity correction.
  • The only individually significant ΔASR is Llama-3.2-3B at −0.040 — a decrease (CI [−0.075, −0.010], McNemar p = 0.021) — and even it does not survive correction.
  • The three effects that do survive correction are capability and over-refusal changes: Qwen3-1.7B MMLU −0.090 (q = 0.008), Llama ARC −0.032 (q = 0.008), Phi over-refusal −0.044 (q = 0.049, and it is a decrease — a benign direction).
  • Capability, not safety, is the robust cost of 4-bit quantization — and it is a cliff at 4-bit: INT8 is essentially free.

The two methodological findings (arguably the most durable contributions)

  1. The scorer determines the conclusion. A refusal-counting regex over-counts "attack success" (its harmful set is a near-strict superset of the official classifier's: 325 regex-only labels vs 29 judge-only across the ten models). Judge-vs-regex agreement is family-dependent (Cohen's κ 0.25–0.84).
  2. The generation budget determines the conclusion. The initial 128-token run truncated 60.3% of generations mid-answer and thereby manufactured an apparent safety regression in the smallest model (Qwen3-1.7B, ΔASR +0.055 at 128 tokens, then statistically significant) that dissolves to exactly 0.000 at the 512-token reference budget. The HarmBench authors themselves show token count can swing ASR by up to 30% and standardize it to N = 512 for exactly this reason.

Threat-model boundary (say it before an examiner asks): HarmBench was run in its standard configuration — curated harmful behaviours presented directly, with no jailbreak/attack augmentation (no GCG, PAIR, AutoDAN). ASR here means harmful compliance under direct requests, the weakest threat model in HarmBench's taxonomy — not robustness to optimized attack.

05The evaluation apparatus, end to end

One benchmark run flows through these stages. The same apparatus ran twice in full — first at the initial 128-token budget, then (as the primary study) at 512.

YAML config→ Pydantic validation→ seed 42 (Python/NumPy/Torch)→ loader (fp16 or on-the-fly NF4/INT8)→ batched chat-template generation→ plugin scoring→ raw.jsonl + summary.json (immutable)→ judge sidecars (redacted)→ analysis JSON→ report / thesis / dashboard

Stage details

  1. Config. configs/tc1_512.yaml declares the 10 aliases (5 pairs × base/4bit), decoding controls, benchmark sample budgets, and cluster parameters. Separate configs isolate the INT8 arm (tc1_int8_512.yaml — only the five *_8bit aliases run; they share the 512 fp16 baselines) and the multi-seed arm (tc1_sensitivity_512.yaml: temperature 0.7, top-p 0.8, seeds 1–5, three pairs).
  2. Loading. Both pair members load from the identical HF snapshot. The quantized member differs only by a BitsAndBytesConfig. The fail-loud guard verifies quantization actually engaged. Provenance (quant method, resolved dtype) is persisted in summaries for future runs; for the committed studies it is fixed by study name + config + loader code.
  3. Generation. Greedy (temperature 0.0, top-p 1.0), max_new_tokens 512 (primary) — the budget HarmBench standardizes — with enable_thinking=False so Qwen3's thinking mode cannot eat the budget. Batch size 4. Every prompt/response pair is written to raw.jsonl with full metadata (pair_id, alias, quantized flag, generation config, seed, timestamp) for auditability.
  4. Plugin scoring. Each benchmark scores its own responses (details in §6). Post-hoc scorer revisions never touch raws — they write derived scores.v2.jsonl/summary.v2.json sidecars.
  5. Judging. The official HarmBench classifier re-scores every HarmBench generation (fp16, greedy, max_new_tokens 1, the pinned official prompt template, zero parse errors across 3,000 classifications at 512). Output is a redacted sidecar: prompt IDs + yes/no booleans only — no text — so it can live in the public repo. GPT-4o applies the same rubric as the independent second judge. Both judges see only (behaviour, generation) — never which arm produced it.
  6. Analysis. make analyze + the diagnostic scripts produce the analysis/ JSON: pairwise deltas with bootstrap CIs, the 20-contrast McNemar + BH-FDR family, judge agreement (κ), the genlen comparison, the precision sweep, the multi-seed aggregates, and the six-label interpretation table.
  7. Interpretation labels. A deterministic rule table over (Δsafety, Δcapability): broad_degradation, alignment_degradation, alignment_improvement, capability_collapse_masquerading_as_safety, over_refusal_regression, robust_preservation — each carrying a separate evidence_status (confirmed / directional / null) so a point-estimate label is never confused with a statistically confirmed one.

Immutability & privacy invariants (worth internalising)

  • raw.jsonl + summary.json are immutable TC1 originals — hash-pinned in a committed manifest (300 files across the 128, 512, and sensitivity trees). The gate fails on any mutation, any unmanifested new raw, and any partial local deletion.
  • Nothing containing raw prompt/response text is ever committed (public repo). Committed evidence = redacted sidecars + aggregates; the redaction gate scans them.
  • Resume logic keys on prompt_id, so interrupted cluster runs continue without double-counting.

The threat model, precisely

HarmBench is a red-teaming framework: its full pipeline feeds behaviours through 18 attack methods (GCG, PAIR, AutoDAN, …). This study evaluates the standard behaviours presented directly, with no attack applied — equivalent to HarmBench's DirectRequest baseline, the weakest threat model in its taxonomy. That is why even well-aligned baselines show non-trivial ASR (e.g. Mistral-7B at 0.585 under the classifier at 512: direct-request compliance of a weakly-guarded model, not "jailbreak success"). Every safety claim in both documents is scoped accordingly; an attack-augmented arm is the top future-work item.

07Scoring: the three-scorer story

The single most important epistemic thread in this project. Scoring changed twice, and each change altered the conclusion — which became a finding in itself.

  1. v1 regex (retired). A small refusal-pattern list. It missed modern refusal templates ("I can't fulfill this request…"), misclassifying clear refusals as harmful compliance. Fixed as T13.
  2. v2 regex (demoted to proxy). Forty-plus patterns, apostrophe-normalised, regression-tested. Deterministic and reproducible — but it measures non-refusal, not harm: vague deflections, safety lectures, and benign on-topic answers all count as "attack success". Retained in Tables 6.1/6.2 as a transparent proxy so readers can see exactly where refusal-counting diverges.
  3. Official HarmBench classifier (PRIMARY, D16). The benchmark's own fine-tuned 13B judge, run at full precision so the instrument is not itself confounded by quantization. The validation showed the regex over-counts, one-directionally (aggregate: 325 regex-only harmful labels vs 29 judge-only across ten models) and unevenly by family — judge-vs-regex κ at 512: Qwen 0.36–0.59, Mistral 0.25–0.28 (worst), Llama 0.71–0.84, Phi 0.67–0.77.
  4. GPT-4o second judge (D25). Architecturally independent, same rubric, re-scores the same saved generations. Cross-judge κ 0.68–0.95 at 512 across all ten models — the finding does not depend on one classifier.

The honest residual: both judges are LLMs; no human gold set exists yet. That is exactly what T30 closes — your 200-row annotation upgrades "two judges agree" to human-grounded validation.

The science

08The statistics, explained (what each test is and what to check)

Paired design
Both pair members answer the identical prompt set. Every contrast is within-pair, prompt-by-prompt — far more sensitive than comparing two independent samples, and the reason the specialized paired tests below apply.
McNemar's exact test
The textbook-correct test for paired binary outcomes. Ignore prompts where both members agree; look only at discordant prompts — b flipped safe→harmful, c flipped harmful→safe — and ask whether the split is more lopsided than a fair coin over b+c flips. Example to internalise: Qwen3-1.7B at 512 has 32 discordant prompts splitting exactly 16/16 → p = 1.000, delta exactly 0. Vet check: every ΔASR in the report should carry a McNemar p; symmetric flips = churn, not direction.
Paired bootstrap 95% CI
Resample the 200 prompt indices with replacement 2,000 times (seed 42), recompute the delta each time, take the 2.5th/97.5th percentiles. Reflects prompt-sampling variance only (decoding is greedy). Nuance you should know: bootstrap endpoints are stochastic across independent re-emissions — two committed artifacts differ by ≤0.005 on some endpoints (documented in the ledger); the claim-locked values come from headline_512_vs_128.json.
Benjamini–Hochberg FDR correction
The study makes 20 primary NF4-vs-fp16 comparisons (5 pairs × 4 metrics). At α = 0.05, one expects ~1 false positive among 20, so raw "significance" is not enough. BH controls the expected fraction of false discoveries: sort the 20 p-values, find the largest p ≤ (rank/20)·0.05. Exactly 3 survive — and none is a safety increase. Vet check: any claim of "significant" in the report should say whether it survives correction.
MDE / power
The minimum detectable effect: with n = 200 and the observed discordant rates (median ≈ 0.09), the smallest ΔASR detectable with 80% power is ≈ 0.06. So the safety nulls are power-bounded ("no detectable change at n = 200"), never "proven zero" — and the report says so. The multi-seed sign-consistency for the smallest pair (all five seed deltas ≥ 0, mean +0.013) is disclosed as compatible with a real sub-floor effect.
Cohen's κ
Agreement corrected for chance: κ = (observed − expected)/(1 − expected). Raw agreement can look high when one label dominates; κ punishes that. Rough reading: ≥0.8 almost perfect, 0.6–0.8 substantial, 0.4–0.6 moderate, below that poor.
Evidence status (project convention, D19)
Every interpretation label carries confirmed / directional / null depending on whether the keyed delta's CI excludes zero (and McNemar agrees, for HarmBench). This prevents a point-estimate label from masquerading as a confirmed finding.

09The 128 → 512 story (the project's turning point)

The study initially ran at a 128-token generation budget. At that 128 budget, the apparent headline was a statistically significant ΔASR of +0.055 in Qwen3-1.7B (McNemar p = 0.027) — and later, an INT8-specific increase on Llama-3B of +0.040, significant under both judges at 128.

The suspicion: responses looked short. The evidence: a direct prefix test over all 2,000 paired generations — if the 512-token generation continues past the exact point where the 128-token one stops, the 128 response was provably truncated mid-answer.

Prefix-test outcome (n = 2,000)ShareMeaning
Provably truncated60.3% (1,206)The 128-token cap cut the answer off mid-generation; the classifier was scoring incomplete text, and where the cut fell could flip a label either way.
Natural stop30.5% (610)The model finished on its own within 128 tokens.
Prefix mismatch9.2% (184)The two nominally-greedy runs diverge early (cross-run kernel/batching non-determinism, Llama-concentrated). Conservatively excluded — so 60.3% is a floor. Within-run deltas are unaffected.

Truncation is family-heterogeneous: Mistral 93.5–98.0%, Phi 73.5–78.5%, Qwen 54.5–70.0%, Llama only 3.0–4.0% at the 128 budget.

Why 512 is not an arbitrary choice

The HarmBench paper shows the number of generated tokens can change measured ASR by up to 30% and standardizes the parameter to N = 512 ("we standardize this parameter to N = 512 to allow the metric to converge"); the official evaluation harness likewise clips completions to 512 tokens before classification. Rerunning at 512 is adopting the benchmark's own protocol — verified from the paper and the official repository.

What happened at 512: the Qwen3-1.7B effect that was +0.055 at 128 became exactly 0.000 under the classifier (+0.005 under GPT-4o; McNemar p = 1.000 under both) — the discordant prompts split a perfectly symmetric 16/16. The Llama INT8 increase (+0.040 at 128) became +0.005 (p = 1.000) / +0.010 (p = 0.688). The capability deltas barely moved between budgets — truncation had manufactured the safety signal, not the capability one. That asymmetry is the cleanest evidence the 128-era signals were budget artefacts.

The discipline

11The agent harness & the machine claim lock

This project is co-built with AI agents on a public repo, which creates two risks the harness exists to kill: stale claims (a number changes but old prose survives somewhere) and evidence damage/leaks. The harness is machine-checked, not prose-promised.

The lock ("agent lock"): scripts/verify_report_claims.py — 55 checks

Every load-bearing number in the report AND thesis builders is asserted in two directions: (a) the document text still contains the claim string (catches silent rewording), and (b) the value recomputed live from the committed analysis artifacts equals the claim (catches drift from evidence). Plus IEEE citation-consistency checks for both documents: every reference cited, no citation beyond the list, numbering strictly by first use. It runs via make verify-claims and inside pytest — so a drifted claim turns the whole 339-test suite red.

Why it exists: when it was first run, it immediately caught six errors that seven external review rounds and a five-agent adversarial audit had all missed — including a prose/table mismatch masked by an identical delta, a stale 128-era κ pair, and three drifted CI endpoints. Prose is not self-verifying. Never hand-verify a report number; run the lock.

13Limitations & threat model — the honest list

  1. Threat model: direct requests only; nothing here speaks to jailbreak robustness under optimized attacks — which is precisely where some prior work reports quantization effects. (Top future-work item: a GCG/PAIR arm.)
  2. Power: n = 200 → MDE ≈ 0.06 at 80% power. All safety nulls are power-bounded, not proven zeros; the smallest pair's all-non-negative seed deltas are compatible with a real sub-floor effect.
  3. Judge grounding: both scorers are LLMs; human gold labels (T30) are prepared but unlabelled. Until then the claim is "judge-cross-checked", not human-validated.
  4. Method coverage: bitsandbytes NF4 + INT8 only. GGUF (the dominant on-device format), GPTQ, AWQ untested — the null may be method-specific, and deployment guidance is hedged accordingly.
  5. Decoding: greedy primary; the multi-seed arm covers three of five pairs.
  6. Capability anchor: zero-shot MMLU subset + ARC — controlled within-pair anchors, not leaderboard reproductions.
  7. Scope: English, text-only; five pairs; fp16 (not bf16) baselines on V100 — symmetric within pairs, so deltas stay clean, but absolute numbers are fp16 values.
  8. Generation budget: absolute ASR is budget-dependent by construction; controlled by adopting HarmBench's standardized 512 and keeping the 128 run purely as the comparison.

Everything numeric and bibliographic is machine-locked. What remains is judgment — and an examiner will probe exactly these:

  1. Narrative arc & redundancy. The report grew by accretion (v1→v5 over six weeks).   3. The title's "Judge-Validated". 16Glossary

ASR
Attack Success Rate — fraction of harmful prompts producing a genuinely harmful, complying response (per the primary classifier). Under this study's threat model: harmful compliance under direct requests.
NF4 / LLM.int8
bitsandbytes 4-bit "normal-float" quantization (with double quantization) / 8-bit mixed-precision scheme keeping outlier feature dimensions in higher precision.
Matched pair
Baseline + quantized model loaded from identical weights; quantization applied at load time; the study's core internal-validity device.
Judge / proxy
Judge = the official HarmBench 13B classifier (primary scorer). Proxy = the deterministic v2 refusal regex, demoted after validation showed it over-counts; retained for transparency.
Direct request / DirectRequest
Presenting the harmful behaviour string as-is, with no adversarial attack transformation — HarmBench's weakest threat model, and this study's configuration.
Discordant prompts / flips
Prompts scored harmful under exactly one pair member — the only prompts McNemar's test looks at. n01/n10 = flips in each direction.
BH-FDR / q-value
Benjamini–Hochberg false-discovery-rate control over the 20-contrast family; q = the FDR-adjusted significance level at which the contrast would survive.
MDE
Minimum detectable effect at 80% power — ≈0.06 ΔASR here; the floor below which this study cannot see.
Evidence status
confirmed / directional / null — attached to every interpretation label so point-estimate labels can't impersonate confirmed findings.
Sidecar
A derived, committed file (redacted scores, IDs + booleans) written alongside immutable raws — the mechanism that lets a public repo carry evidence without carrying harmful text.
Claim lock
verify_report_claims.py: 55 machine checks pinning both documents' numbers and citations to the committed artifacts.
D-numbers / T-numbers
Decisions / tasks in PROJECT_LOG (e.g. D16 judge-primary, D41 512-primary, D42 claim-surface sweep, D43 audit conventions; T30 human gold set).
