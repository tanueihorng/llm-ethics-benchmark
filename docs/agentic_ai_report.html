    00 — For the supervisor

    Executive summary

    This is a separate, standalone report. It does not change or restate the scientific findings of the project — those live in the main FYP report and thesis. Its single purpose is to explain, in accessible terms, how the work was produced: the agentic AI tooling I built and operated to do a rigorous research project faster, and the safeguards that keep the results trustworthy.

    The one thing to take away

    I did not ask an AI to "do my project." I used AI coding agents the way an engineer uses power tools: to execute my decisions quickly. I then built a verification harness around those tools — a set of automatic checks, immutable evidence, and independent review — that repeatedly caught the AI's mistakes. The most important scientific correction in this project (§9) happened because of that harness, not in spite of the AI. The trustworthiness of the results rests on human judgement plus machine-checkable rigor, not on trusting a model.

    01 — Plain-English primer

    What "agentic AI" actually is

    If you have used ChatGPT, you have used a language model: you type, it types back. An agentic AI tool is the next step — the same kind of model, but wired up so it can take real actions on a real project. Three rungs:

    i
    In plain termsA language model alone is a clever autocomplete. An agent can press the buttons. A harness is the workshop I built around the agent — with safety rails, a logbook, and a final inspection — so that a fast but occasionally-wrong assistant can work on a real scientific project without breaking it or quietly corrupting the data.

    The catch — and the reason the rest of this report matters — is that these tools are fast but fallible. They can write a hundred lines of correct code in seconds and then state something subtly wrong with total confidence. An agentic workflow is only as trustworthy as the checks wrapped around it. My contribution to this project's method was building those checks.

    03 — The human-in-the-loop

    How the work actually happens

    Every change moves through the same four-beat loop. I open and close it; the agent only ever does the middle. Nothing an agent produces is believed until it has been verified and recorded.

    Who owns what

    The honest division of labour. The left column is irreducibly human — it is the research. The right column is what the tooling absorbed.

    I own (the science)

    • The research question and hypotheses
    • The matched-pair methodology & controls
    • Which models, benchmarks and metrics
    • Interpreting results & what counts as significant
    • Judgement calls (e.g. promoting the judge model to primary)
    • The scientific narrative and every claim in the report
    • Communication with my supervisor

    The agent accelerates (the execution)

    • Writing boilerplate: loaders, plugins, SLURM scripts
    • Refactoring and keeping code consistent
    • Generating and maintaining the test suite
    • Searching the codebase and drafting documentation
    • Catching inconsistencies across files
    • Running adversarial self-audits I commission
    • Keeping the logbook and handoff notes in sync

    The dividing line: the agent proposes; I dispose. No result, label, or conclusion enters the project until it has cleared verification and been recorded — which is exactly what §4–§9 enforce.

    04 — The heart of it

    The agent harness: an operating system for AI assistants

    A coding agent starts every session with no memory of the last one. Left alone it will re-learn the project from scratch, contradict earlier decisions, and occasionally touch something it shouldn't. The harness is six thin layers — all of them just files in the repository — that orient a fresh agent, scope it, check it, and protect the evidence.

    i
    In plain termsThink of onboarding a brilliant but forgetful new assistant every single morning. The harness is the printed induction pack on their desk: the rules, the project logbook, the one-page brief for today's task, a colleague who double-checks their work, a few safety interlocks, and a sign-off checklist they can't skip. It is the difference between a helpful assistant and a hazard.

    Crucially, every layer is a plain file under version control — diffable, reviewable, and auditable by you. There is no hidden automation. If you want to see exactly what an agent is told, you can open the file.

    05 — Memory & discipline

    One logbook the whole project trusts

    The biggest risk with a forgetful, fast assistant is drift — yesterday's decision quietly contradicted today. The cure is a single, authoritative logbook that every change must update. In this project that is docs/PROJECT_LOG.md, and the rule is absolute: nothing is committed without a matching log entry.

    What makes the logbook trustworthy

    append-only

    Decisions are never edited

    When a decision is reversed, a new dated entry references the old one — so the reasoning trail is tamper-evident, not rewritten. There are 37 such decisions (D1–D37).

    one row per change

    The log is part of the change

    Columns: when · files · change · why · was the report rebuilt · who. The git history shows nearly every feature commit paired with a log row.

    stable references

    Tasks & decisions are numbered

    Open items are T<N>, decisions D<N>, never renumbered — so any conversation or commit can point to one unambiguously.

    machine-enforced

    It can't quietly fall behind

    The finish gate fails if the working tree changed but the log didn't, and a scanner flags out-of-date "current" text in the bridge notes.

    i
    In plain termsThis is a lab notebook with the discipline turned up. Because two different AI agents and I all read and write the same logbook — and a check refuses any change that skips it — there is exactly one place to learn "where the project stands," and it is always current.

    07 — Why the results stay trustworthy

    Guardrails that keep AI-accelerated research honest

    Speed is worthless if it quietly corrupts the evidence. Three guardrails — immutable evidence, a redaction boundary, and end-to-end reproducibility — mean an AI can move fast through this project without being able to fake, mutate, or lose a single result.

    guardrail 1

    Immutable evidence

    The 120 raw result files from the cluster are locked by SHA-256 hash and never edited. Every metric is re-derived from them by script. Any silent change is caught by the finish gate.

    guardrail 2

    Redaction boundary

    Committed safety-judge files contain only prompt IDs and yes/no labels — never the raw harmful prompts or model responses. The study reproduces from the public repo without redistributing harmful text.

    guardrail 3

    Reproducibility

    Fixed seeds (42), greedy decoding, pinned dependencies, offline cluster runs, and a 306-test suite. Re-running the analysis re-creates the published numbers byte-for-byte.

    Corrections without touching the originals

    When a scoring method had to change (and it did — see §9), the original cluster outputs were not overwritten. The corrected scores were written into separate, auditable sidecar files. So the project carries both the original record and every correction, side by side, with a full trail of which is which. This is the single most important reason the headline result can be reproduced exactly from immutable files today.

    An honest limit, stated plainly

    No guardrail is perfect, and the harness says so itself. The redaction scanner catches explicit leak patterns and data-shaped snippets, but it cannot prove that no paraphrase of a sensitive response exists anywhere — so it is described in the project's own self-evaluation as a strong filter, not a proof. Significance is reported as nominal and uncorrected, with the multiple-comparisons caveat disclosed. Stating limits like these, rather than hiding them, is part of what makes the rest credible.

    08 — The sign-off checklist

    One command that must pass: make agent-check

    Before any change is considered "done," a single command runs eight mechanical checks. If any one fails, the change is not finished. This is the moment where the project's prose rules become executable — a checklist an agent cannot talk its way around.

    ✓
    Docs in sync — the two rulebooks (AGENTS.md, CLAUDE.md) must stay byte-identical; the build fails if they ever drift.
    ✓
    Logbook updated — if the working tree changed but PROJECT_LOG.md didn't, fail. The log is part of the change.
    ✓
    Raw evidence intact — re-hash all 120 raw files against the manifest; fail on any mutation or unrecorded new file.
    ✓
    No data leaks — scan docs for raw harmful prompt/response text or sensitive data keys; fail if any slipped in.
    ✓
    No stale claims — scan for superseded "current-state" phrases (e.g. "results pending") that should have been updated.
    ✓
    Report fresh — if a result-bearing file changed without the report being rebuilt, fail. Editing the report source alone won't satisfy it.
    ✓
    Clean diff — fail on stray whitespace or leftover merge-conflict markers.
    ✓
    All tests pass — run the full 306-test suite; fail if any test fails.
    i
    In plain termsThis is the pre-flight checklist. A pilot doesn't take off on vibes; they run the list and every item must read green. make agent-check is that list for this project — and a second copy (make harness-eval) deliberately breaks the repo in known ways to prove the checklist actually catches problems.
    09 — The proof it isn't a rubber stamp

    When the harness caught the AI — and changed the science

    This is the most important section for judging trust. If an AI-accelerated project never disagreed with the AI, you should be suspicious. This one disagreed repeatedly — and the disagreements were caught, verified, and used to correct the findings. Here are the moments that matter.

    "Did the model get safer — or did my scorer just mislabel it?"
    decision D16 — the correction that reset the headline result

    The headline example (D16). My first, fast safety scorer was a pattern-matcher. It worked, but I did not trust it. I ran the official 13-billion-parameter HarmBench classifier as an independent judge over the same outputs. It materially disagreed with my fast scorer — the pattern-matcher was over-counting harmful answers. The disagreement was not cosmetic: it moved the study's only statistically significant safety result from one model to another, and flipped the direction of the smallest model. I promoted the judge to the primary scorer and rewrote the analysis. Because the raw evidence was immutable, the correction was made safely in sidecar files — no original was touched.

    That is the whole thesis of this report in one episode: the value was not the AI's first answer — it was the verification environment that proved the first answer wrong and let me fix it without losing the original record. The same pattern recurred:

    A confident "story" did not survive scrutiny

    An appealing mechanism explanation — "4-bit quantization erodes the model's refusals" — was tested by a dedicated adversarial critic pass. It failed: the effect was symmetric (it moved both ways) and traced to general capability softening, not a targeted attack on safety. The overclaim was caught and not written; the report states the honest, weaker finding.

    10 — What the acceleration bought

    The payoff, measured honestly

    The acceleration did not show up as "the AI wrote my project." It showed up as depth I could afford: a study that grew from a basic 3-pair experiment into a 5-family, 3-precision investigation with two independent judges, two capability benchmarks, a mechanism probe, and two full audits — each guarded by tests.

    0 families
    grown from 3 model pairs to 5 across 4 model families — feasible only because the routine scale-up was automated
    0 judges · 0 benchmarks
    independent cross-checks on both the safety and the capability axes

    The point is not "more is better." It is that each robustness layer in §9 — the second judge, the multi-seed run, the second benchmark, the audits — is exactly the kind of careful, repetitive work that usually gets skipped under time pressure. The tooling made that work cheap enough to actually do, and the harness made it safe to do without destabilising the result. The acceleration was spent on rigor, not on cutting corners.

    11 — Responsible use

    Limits, and how I stayed responsible

    Using AI tooling responsibly means being clear-eyed about what it can't do, and leaving an audit trail so a skeptical reader can check the work rather than take my word for it.

    limit

    Models can be confidently wrong

    Every load-bearing claim is verified against committed evidence, an independent judge, or a test — never accepted on the model's say-so. §9 is the receipts.

    limit

    An agent forgets between sessions

    Mitigated by the single-source-of-truth logbook and generated handoff notes — but those notes are explicitly marked "verify before acting."

    limit

    Automated checks aren't proofs

    The redaction scanner is a strong filter, not a guarantee; significance is nominal/uncorrected. Both are disclosed in the project's own write-ups.

    limit

    Acceleration ≠ authorship

    The AI accelerated execution. The research question, design, interpretation and conclusions are mine, and are declared as such in the thesis's AI-usage statement.

    Two practices keep this honest and checkable: first, the project is fully auditable — every decision, with its rationale and date, is in the append-only log, and every result re-derives from hash-locked evidence. Second, AI usage is declared, not hidden: the main thesis carries an explicit AI-usage statement, and this report exists precisely to make the method transparent to you.

    The standard I held myself to

    A reader who distrusts AI entirely should still be able to trust this project — by ignoring the tooling and checking the evidence directly. The immutable raw files, the independent judge results, the test suite, and the dated decision log are all there to be inspected. The tooling made the work faster; the evidence is what makes it true.
