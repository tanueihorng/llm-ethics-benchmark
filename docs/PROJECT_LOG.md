### D33. 2026-06-15 - Mechanism study (refusal-margin erosion under quantization): harness built + design de-risked; TC1 capture pending

**Why:** the empirical results (D16/D32) are a rigorous null plus one modest, borderline effect (Qwen 1.7B). To move the project from "what happens" to "why" — the lever that most raises it from benchmarking to research — we add a *mechanism*: does NF4 erode the model's refusal **decision margin**, and do behavioural flips concentrate where that margin was already thin (explaining why the smallest model moves first; Proskurina et al. 2024 arXiv:2405.00632 — quantization moves low-confidence samples most)?

**Design (de-risked by a 3-agent design workflow; critic verdict GO_WITH_CHANGES):** the naive "first-token refusal-sign flips where behaviour flips" metric is **circular** under greedy decoding (the sign already determines the judged label) and **wrong-locus** for Mistral (soft-preamble-then-comply decides late). Adopted instead: a **continuous, scale-invariant** margin `m = logsumexp(z[refusal_set]) − logsumexp(z[compliance_set])` analysed as the paired shift **Δm across all 200 HarmBench prompts** (Wilcoxon + bootstrap), NOT flip-only. Required controls (all built/planned): (1) a **capability/confidence confound control** — the same precision-induced entropy/decisiveness shift on neutral MMLU/XSTest tokens, which the refusal-margin shift must exceed (else it is "NF4 lowers confidence everywhere", a capability story); (2) a **sequence-level margin** teacher-forced over the committed response (min/mean) to capture late-resolved refusals; (3) an **fp16-vs-judge AUC gate** — the margin must predict the baseline judge label before any quantization claim; (4) flips treated as a **confirmatory zoom**, and "smallest moves first" tested by a pooled logistic `flip ~ baseline_margin + family` (n≈1000) since per-pair flip subsets (Llama 4, qwen_4b 9) are underpowered and size is confounded with family.

**Built this commit (branch `mechanism-refusal-margin`, run pending):** `ethical_benchmark/analysis/refusal_margin.py` (pure, CPU-testable metric core); `scripts/capture_refusal_margin.py` (TC1 GPU entry — teacher-forced forward passes, redacted `scores.margin.<precision>.jsonl` sidecars, NO generation, never touches raw/summary/judge artifacts; precision-agnostic so the same code does fp16/NF4/INT8); `slurm/capture_refusal_margin.sbatch`; `tests/test_refusal_margin.py` (15 tests, math + token-set + redaction). Full suite **261 pass**; CPU functional smoke on a tiny model confirms the forward-pass path. Lit anchors: Arditi et al. 2024 (refusal direction, arXiv:2406.11717), Egashira et al. 2024 (arXiv:2405.18137), Ashfaq et al. 2021 (quantization/calibration, arXiv:2111.08163). Tracked as T28.

**RESULTS + honest reframe (2026-06-15; capture TC1 job 61136 over all 10 aliases + Mac-side `scripts/refusal_margin_analysis.py`; adversarial verification = recompute PASS, interpretation critic SOUND_WITH_CAVEATS):** the clean "NF4 erodes refusal" story did NOT survive verification. The honest finding is **boundary instability, not targeted alignment erosion.** (1) *Gate:* the first-token margin predicts the baseline judge label for four families (AUC 0.86–0.90) but only weakly for Mistral (0.69 — decides late, refusal-shaped preamble then complies), which is flagged. (2) A thin baseline margin predicts flips, but the within-family (z-scored) AUC is only **0.64** — the naive pooled 0.76 is ~half a between-family/ecological effect (n≈5), and within-pair it is near chance for the two thinnest families (Qwen-1.7B 0.61, Mistral 0.54). (3) Flips are **symmetric** (50 harmful-ward / 42 safe-ward); a thin margin predicts safe-ward flips (AUC 0.78) at least as strongly as harmful-ward (0.75) → near-definitional boundary fragility, NOT one-directional safety erosion. (4) The margin shift is significant (Wilcoxon p<0.001, n=200) but **model-specific in direction** (Qwen comply-ward +1.2/+4.0; Llama/Phi refuse-ward −1.1/−0.3) and **first-token-specific** (whole-sequence margins reverse the sign for Qwen-1.7B and Phi). Qwen-4B erodes MORE (+4.0) yet flips fewer (9 vs 21) because its baseline margin is far wider (median 13.0 vs 3.8) — the mechanistic reason the 4B is behaviourally intact. (5) *Confound control:* for the flip-driving pair **Qwen-1.7B the shift is consistent with GENERIC confidence/capability softening** (neutral-token entropy +0.21 ≫ harmful +0.02), NOT a targeted alignment shift — so the small-model wobble reads as **capability-driven boundary instability**, converging with the significant ΔMMLU, the multi-seed fragility, and the ARC story. This SUPPORTS the study's capability-vs-alignment dichotomy (on the capability side) rather than adding an alignment-erosion claim. Folded honestly (with explicit caveats) into report **§6.14** + Ch9 follow-ups + Ch10 + Abstract; the overclaimed "targeted erosion" framing was caught by the adversarial pass and NOT written. Optional follow-ups: Arditi activation-direction probe (independent window), a paired neutral-margin control with a significance test, and an INT8 precision point (fp16→INT8→NF4 trend).

### D1. 2026-05-23 — Drop Qwen 0.8B; finalise the three-pair design

**Decision:** Study uses three pairs: Qwen 2B, Qwen 4B, and Llama 3.2 3B (cross-family). Qwen 0.8B is dropped.

**Rationale:** At 0.8B, the BF16 baseline itself is at or barely above MMLU chance (~25%), making the alignment-versus-capability disambiguation unreliable (the capability anchor degenerates). Removing it eliminates the largest scientific risk in the study. The 2× scale gap (2B → 4B) is sufficient to test scale sensitivity within the compact-deployment regime, even though the original 5× gap (0.8B → 4B) would have been more dramatic. The scope is reframed from "scale-sensitivity across small-to-compact models" to "quantization effects in the compact-deployment regime (2B–4B)."

### D2. 2026-05-23 — On-the-fly NF4 quantization from identical baseline weights

**Decision:** Both members of every pair load from the same Hugging Face `model_id`. The only difference is whether `BitsAndBytesConfig(load_in_4bit=True, bnb_4bit_quant_type="nf4", ...)` is injected at `from_pretrained` time.

**Rationale:** Eliminates publisher- and conversion-asymmetry as confounds. Earlier plan used a pre-quantized `*-bnb-4bit` checkpoint, which conflated quantization effect with the specific publisher's NF4 pipeline. On-the-fly quantization is the strongest practical isolation of quantization as the experimental variable. Loader (`models/loader.py:_build_bnb_4bit_config`) and config schema both updated to support this. Test `test_models.TestPromptFormatting` confirms `enable_thinking=False` is also handled correctly.

### D3. 2026-05-23 — Group SLURM jobs by model, not by benchmark

### D11. 2026-05-28 — Refusal classifier revision v1 → v2; rescore all raw outputs in place

### D14. 2026-06-06 — Judge classifier runs in full precision (fp16), not quantized

**Decision:** The HarmBench judge classifier (`cais/HarmBench-Llama-2-13b-cls`) runs in float16 (full precision) by default for the validation. The runner exposes `--precision {fp16,8bit,4bit}` defaulting to `fp16`; the sbatch requests fp16, probes and logs the actual GPU + VRAM at startup, and records the GPU/precision used in `summary.judge.harmbench_cls.json`. If the 13B classifier does not fit in fp16 on the available GPU (i.e. TC1's V100 is the 16 GB variant, ~26 GB needed for fp16), the documented fallback is `--precision 8bit` (~13–14 GB, far closer to full precision than the 4-bit NF4 used on the target models), reported explicitly. 4-bit is the last resort.

**Rationale:** The judge is the *reference instrument* used to validate the (4-bit-quantized) target models. Quantizing the judge itself with the same NF4 scheme would introduce exactly the confound the check exists to rule out — any judge–v2 disagreement could then be blamed on judge quantization rather than being informative about the target models. Full precision keeps the validation instrument clean. Logging the realised precision and hardware preserves reproducibility and makes any fallback transparent in the report (§6.12) rather than hidden. This reverses the original sbatch choice (NF4 "to fit the V100"), which optimised for fitting over fidelity; for a validation instrument, fidelity wins. Superseded the NF4 wording in §6.12 and the sbatch.

If fp16 still OOMs on TC1, the first operational response is to reduce batch size further (2 or 1). Only after that should the job switch to the documented `--precision 8bit` fallback, and any fallback must be reported in the judge summary and §6.12.

### D16. 2026-06-06 — Official HarmBench classifier supersedes the v2 regex for HarmBench ASR; refine alignment_degradation rule

**Decision:** The official HarmBench classifier (`cais/HarmBench-Llama-2-13b-cls`, fp16) is the **primary** scorer for HarmBench Attack Success Rate in all headline conclusions. The v2 refusal regex is demoted to a **secondary, refusal-based non-refusal-rate proxy**, retained in the repo and report for transparency. Scope is HarmBench-only: XSTest over-refusal stays v2-scored (the classifier does not judge over-refusal) and MMLU is unaffected. Separately, `classify_pair_change` is refined so `alignment_degradation` requires `harm_delta >= harm_tol AND capability_delta > -cap_tol` (capability preserved); ASR-up-with-capability-down now correctly falls through to `broad_degradation`.

**Rationale:** The judge validation (T20, job 61047, fp16 on a 32 GB V100, n=200×6, 0 parse errors) showed the v2 regex materially disagrees with the benchmark's own classifier — agreement is poor for Qwen (Cohen's κ ≈ 0.19–0.37) and high for Llama (κ ≈ 0.69–0.79). The regex equates "non-refusal" with "attack success" and therefore over-counts ASR, most severely for Qwen, which produces much non-refusal-but-not-harmful text. The disagreement is conclusion-level: under the judge the only statistically significant ΔASR moves from Qwen 4B (proxy: +0.065, CI [+0.025,+0.110]) to Qwen 1.7B (judge: +0.055, CI [+0.010,+0.100]), and the small-model direction flips from apparently-negative to significantly-positive. Keeping a scorer demonstrably over-counting ASR 3–4× on Qwen as the headline metric would be indefensible; the HarmBench paper specifies this classifier. The rule refinement enforces the report's own definition of alignment_degradation ("ASR up with capability preserved") — without it, Qwen 1.7B under the judge (ΔASR +0.055 significant, ΔMMLU −0.087 significant) would be mislabelled alignment_degradation rather than the honest broad_degradation. Verified the refinement changes no v2 pair labels.

**Impact (judge-primary, refined rule):** Qwen 1.7B → broad_degradation (significant ΔASR +0.055 and significant ΔMMLU −0.087 — the study's confirmed worst case); Qwen 4B → alignment_degradation but directional only (ΔASR +0.025, CI touches zero); Llama 3B → broad_degradation (ΔASR ≈ 0, significant ΔMMLU −0.043). Under the official classifier, NF4 never reduces true harmful compliance in any pair. The judge validation thus demonstrates the framework's diagnostic value by overturning a regex artefact and relocating the one robust safety regression to the smallest model. Implemented in `compare_quant_pairs.classify_pair_change`, `scripts/judge_agreement.py` (now emits judge bootstrap CIs + judge-primary labels), tests (`test_quant_analysis.py`, +2), and the FYP report (Tables 6.1/6.2/6.3, §6.1–§6.12, Abstract, Ch10).

### D10. 2026-05-28 — Add alignment_improvement label to interpretation taxonomy

**Decision:** Extend `classify_pair_change` from four labels to five by adding `alignment_improvement`, which fires when `ΔASR ≤ −harm_tol AND |ΔOR| < refusal_tol AND ΔMMLU > −cap_tol`. The label is the symmetric mirror of `alignment_degradation` and is checked after the capability-collapse and alignment-degradation conditions but before `robust_preservation`.

**Rationale:** The original 4-label taxonomy was asymmetric on the safety axis: it had `alignment_degradation` for the bad direction (ASR up with capability preserved) but no mirror label for the good direction (ASR down with capability preserved). Qwen 4B (ΔASR=−0.045, ΔMMLU=−0.004, ΔOR=0.000) is the textbook case of a capability-preserving harmful-compliance reduction — a genuinely desirable outcome — but was falling through to the `broad_degradation` fallback, which inverts the scientific reading. Adding the symmetric label fixes Qwen 4B's classification without re-categorising Qwen 2B (still capability_collapse) or Llama (still alignment_degradation). The taxonomy is now symmetric around the safety axis: improvement and degradation are first-class outcomes, capability-collapse remains the sentinel for fake safety wins, and broad_degradation becomes a true fallback for mixed signals.

## 6. Maintenance contract (the rules that keep this repo coherent)

These are the standing conventions. They apply to every agent session (Claude, Codex, Cursor, Aider, or a human) and to every commit.

1. **Every change writes one row to the §4 changelog table.** Columns: `When (UTC+8) | Files | Change | Why / Notes | Report? | Who`. ## 7. How to use this log

**When you (or a future Claude session) make a change to the repo:**
1. Update the relevant section here in the same change. Don't push code without a log entry.
2. If the change is just a file edit (typo, small refactor), add one line to the changelog.
3. If the change involves a real decision (choosing approach A over B, dropping scope, adopting a new convention), add an entry to §3 (Decisions) with the rationale.
4. If the change creates work for later, add to §2 (Open Actions).
5. If the change affects current state (e.g., experiments now running, environment broken, quota exhausted), update §1 (Status snapshot) in place.

1. Tick `[x]` next to the item in §2.
2. Move it (with a one-line note) into the latest changelog entry, or create a new dated changelog entry if it's substantial.

**When you make a decision:**
1. Append to §3 with date, decision, and rationale.
2. Never edit prior decisions. If reversing one, add a new entry that references the old.
