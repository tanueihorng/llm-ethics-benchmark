2. Decision Register

This section lists the choices that should be locked before large experiment runs. These are the places where method, code, and compute capacity can diverge if not handled deliberately.

ID Decision Current repo position Recommendation Why it matters Latest safe decision date
D1 Use HarmBench only, or add JailbreakBench? HarmBench only. No JailbreakBench plugin exists. Use HarmBench only. Remove "or JailbreakBench" from the proposal unless there is explicit time to implement a second harmful-prompt benchmark. Adding JailbreakBench increases implementation, validation, and report burden. 19 May 2026
D2 Use LLM-as-judge for HarmBench? Current HarmBench scoring is heuristic: malformed check plus refusal-pattern detection. Use LlamaGuard-style judging for final HarmBench if TC1 is stable. Keep heuristic scoring as a fallback or ablation. LLM judge makes harmful compliance classification more defensible, but adds implementation and compute. 24 May 2026
D3 Should XSTest use an LLM judge? Deterministic refusal parser. No. Keep deterministic parsing for XSTest. XSTest is specifically about false refusal on benign prompts; deterministic refusal parsing is cheap, reproducible, and adequate for a first controlled study. 19 May 2026
D4 Should MMLU use an LLM judge? Exact multiple-choice parser. No. Keep exact answer matching. MMLU has gold labels. A judge would add noise and compute without improving the metric. 19 May 2026
D5 Baseline precision policy. dtype: auto, which resolves differently by device. Use explicit dtype for final runs. Prefer bf16 for Qwen baselines if TC1 supports it, and document any exception. Without this, "baseline" differs by hardware and can weaken the quantization comparison. 24 May 2026
D6 Thinking mode control. Not explicitly controlled in generation code. Add explicit config support. Pass enable_thinking=False where supported by tokenizer chat templates. Thinking output can change length, formatting, refusal behavior, and parser reliability. 24 May 2026
D7 Keep Llama as main result or secondary result? Llama pair is in the default matrix and README. Tier the scope. Qwen four-model scale study is must-have. Llama pair is cross-family extension if TC1 access and time are healthy. RQ5 depends on Llama, but the August report can still be meaningful with Qwen preliminary findings if time compresses. 31 May 2026
D8 Use MacBook results in final tables? Repo is CUDA-oriented for quantized loading. No. Use Mac only for development and small smoke checks. Use TC1 for final numbers. Different devices can change dtype, kernels, quantization backend, speed, and reproducibility. Immediate
D9 Hugging Face gated access. HarmBench access may require authentication on TC1. Resolve early. Authenticate TC1 and verify dataset/model access before any large run. This can block the primary safety benchmark entirely. 24 May 2026
D10 TC1 GPU stack. Previous notes reported CUDA-driver mismatch risk. Verify with a SLURM CUDA smoke job before experiments. Do not start large matrix runs until this is clean. CPU fallback can make runs too slow and inconsistent with the intended environment. 31 May 2026

10. Recommended Methodology Statement

The following wording is currently the cleanest version for the project, assuming the LLM judge is adopted for HarmBench:

This study evaluates harmful compliance, over-refusal, and general capability across matched baseline and 4-bit quantized small language model pairs. HarmBench is used for harmful compliance, XSTest for benign over-refusal, and an MMLU subset for general capability. All final experiments are run in a controlled TC1 environment using fixed prompts, decoding settings, and output schemas. HarmBench responses are scored using a fixed safety classifier such as LlamaGuard, while XSTest and MMLU use deterministic scoring. The Qwen 0.8B and 4B pairs form the primary scale-sensitivity study, and the Llama 3.2 3B pair is used as a secondary cross-family comparison where compute permits.

If the LLM judge is not adopted, replace the HarmBench sentence with:

HarmBench and XSTest responses are scored using deterministic refusal and malformed-output parsing, with the limitations of heuristic safety classification explicitly discussed.
