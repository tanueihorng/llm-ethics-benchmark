5. Issues, Bugs, Mismatches, and What They Mean

Priority Issue Evidence / explanation Impact Suggested fix
High HarmBench evaluator does not match the report The report says HarmBench should use LlamaGuard classification, but the current plugin uses refusal heuristics: see ethical_benchmark/benchmarks/harmbench.py and ethical_benchmark/benchmarks/utils.py. Your primary safety metric is methodologically inconsistent with the written FYP design. Implement a judge-based HarmBench scorer (LlamaGuard or an approved replacement), or explicitly revise the report if you want to keep heuristic scoring.
High Repo scope still includes Llama although the report is now Qwen-only configs/tc1.yaml, README.md, docs/methodology.md, and cross-family analysis logic still assume Qwen + Llama. Easy to accidentally run or report out-of-scope experiments. Create a Qwen-only primary config and update default docs/analysis language to reflect the refined report scope.
High BF16 baselines are not actually enforced The report says BF16 baselines, but config uses dtype: auto, and the loader maps CUDA auto to FP16 rather than BF16. Baseline-vs-quantized comparisons are not running under the exact precision policy described in the report. Set the baseline Qwen entries to explicit dtype: bf16 and add tests so this cannot drift.
High Non-thinking mode is stated in the report but not explicitly applied in code The report calls for enable_thinking=False, but the chat-template call currently omits it. Could change response structure, scoring behavior, and comparability. Extend generation config and tokenizer template calls to explicitly pass enable_thinking=False where supported.
High CUDA driver mismatch on TC1 Job 59039 reported device count: 1 but cuda available: False, with a warning that the NVIDIA driver is too old for the installed Torch build. Current TC1 runs are effectively CPU fallback even when a GPU is allocated. Create a new remote env with a Torch build compatible with the TC1 driver, likely CUDA 12.1 or 11.8.
Medium HarmBench is gated on Hugging Face Job 59037 failed with DatasetNotFoundError because walledai/HarmBench requires authentication. Primary benchmark runs cannot proceed until access is resolved. Authenticate the TC1 environment with a Hugging Face token and ensure access to the gated dataset is approved.
Medium Benchmark configs do not currently match the report The report describes different splits/datasets/sample sizes than the current config, including a different XSTest source and smaller report-controlled sample counts. Results produced by the repo would not perfectly match the methodology chapter. Make the config match the report exactly, or explicitly revise the report to the implemented design before running final experiments.
Medium Fresh environment dependencies are incomplete for the full study path accelerate was missing and had to be installed manually; bitsandbytes is still not in requirements.txt. Fresh env setup is fragile and the 4-bit path is not fully reproducible from one install command. Add the missing runtime dependencies to the environment spec or provide an explicit TC1 setup file.
Low Ad hoc prompt test command in terminal log was partially corrupted The heredoc capture in the log contains inserted garbage text, so that exact snippet is not a clean reproducibility artifact. The test still showed that model loading and text generation worked, but it should not be treated as a polished benchmark harness. Replace it with a clean checked-in smoke script or a tiny generated SLURM probe.
